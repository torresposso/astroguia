# Mastra Project Structure & File-Based Agents

Research for GitHub issue #4: "Estructura de proyecto Mastra y agentes file-based"

## Summary

Mastra is a TypeScript framework for building AI agents. It supports two agent definition modes:
1. **Code-based**: Instantiate `Agent` class directly in TypeScript files
2. **File-based** (beta, `@mastra/core@1.50.0+`): Convention-over-configuration using directories under `src/mastra/`

File-based agents are discovered by the Mastra bundler during `mastra dev` / `mastra build`. They reduce glue code and are especially useful when coding agents (human or AI) need to navigate the project. Code-based and file-based agents can coexist in the same project.

## Initialization Command

```bash
# npm
npm create mastra@latest

# pnpm
pnpm create mastra

# yarn
yarn create mastra

# bun
bunx create-mastra
```

Key CLI flags for `create-mastra`:
- `--default` — quickstart with defaults (src/, OpenAI, no examples)
- `--template <slug>` — scaffold from a template (see https://mastra.ai/api/templates.json)
- `--components agents,tools,workflows` — which folders to scaffold
- `--llm openai|anthropic|groq|google|cerebras`
- `--llm-api-key <key>` — write to `.env`
- `--example` / `--no-example` — include or skip example code
- `--dir <path>` — target directory for Mastra source (default: `src/`)

For existing projects, use `mastra init` with similar flags.

## Default Project Structure (generated by `create-mastra`)

```
my-mastra-project/
├── src/
│   └── mastra/
│       ├── agents/
│       │   └── weather-agent.ts       # Example code-based agent
│       ├── tools/
│       │   └── weather-tool.ts        # Example tool
│       ├── workflows/
│       │   └── weather-workflow.ts    # Example workflow
│       ├── scorers/
│       │   └── weather-scorer.ts      # Example scorer
│       └── index.ts                   # Mastra entry point
├── .env.example                       # Template for env vars
├── package.json
└── tsconfig.json
```

Mastra is **unopinionated** about structure beyond the conventions it recognizes.

## Minimal File-Based Agent Structure

```
src/
└── mastra/
    ├── agents/
    │   └── weather/                  # Agent directory = agent ID
    │       ├── config.ts             # Required: model & runtime options
    │       ├── instructions.md       # Required: system prompt
    │       ├── tools/                # Optional: typed functions
    │       │   └── get-weather.ts
    │       ├── skills/               # Optional: reusable instruction snippets
    │       │   └── forecast-review.md
    │       ├── memory.ts             # Optional: memory configuration
    │       ├── workspace.ts          # Optional: workspace configuration
    │       ├── processors/           # Optional: I/O processors
    │       ├── scorers/              # Optional: evaluation scorers
    │       └── subagents/            # Optional: child agents
    ├── workflows/                    # File-based workflows
    ├── storage.ts                    # Global storage config
    ├── observability.ts              # Global observability config
    ├── logger.ts                     # Global logger config
    ├── server.ts                     # Server configuration
    ├── studio.ts                     # Studio configuration
    └── index.ts                      # Mastra entry point (minimal with file-based)
```

## How Agents and Workflows Coexist

Agents and workflows serve different purposes but interoperate:

- **Agents**: Open-ended tasks. The LLM decides which tools to call and when to stop. Use `Agent` class or file-based agent directories.
- **Workflows**: Predetermined multi-step processes with explicit control flow. Use `createWorkflow()` + `createStep()` from `@mastra/core/workflows`, or file-based workflow directories.
- **Composition**: Workflows can call agents as steps. Agents can invoke workflows. Both register on the same `Mastra` instance.

### Code-Based Agent Example

```typescript
// src/mastra/agents/weather-agent.ts
import { Agent } from '@mastra/core/agent'
import { weatherTool } from '../tools/weather-tool.ts'

export const weatherAgent = new Agent({
  id: 'weather-agent',
  name: 'Weather Agent',
  instructions: 'You are a helpful weather assistant...',
  model: 'openai/gpt-5.5',
  tools: { weatherTool },
})
```

### Tool Definition (shared, code-based)

```typescript
// src/mastra/tools/weather-tool.ts
import { createTool } from '@mastra/core/tools'
import { z } from 'zod'

export const weatherTool = createTool({
  id: 'get-weather',
  description: 'Get current weather for a location',
  inputSchema: z.object({ location: z.string().describe('City name') }),
  execute: async ({ location }) => {
    return { location, temperatureCelsius: 21, conditions: 'sunny' }
  },
})
```

### Workflow Example

```typescript
import { createWorkflow, createStep } from '@mastra/core/workflows'
import { z } from 'zod'

const step1 = createStep({
  id: 'step-1',
  inputSchema: z.object({ message: z.string() }),
  outputSchema: z.object({ formatted: z.string() }),
  execute: async ({ inputData }) => {
    return { formatted: inputData.message.toUpperCase() }
  },
})

export const testWorkflow = createWorkflow({
  id: 'test-workflow',
  inputSchema: z.object({ message: z.string() }),
  outputSchema: z.object({ output: z.string() }),
})
  .then(step1)
  .commit()
```

## The Mastra Class — Tying Everything Together

The `Mastra` class in `src/mastra/index.ts` is the central configuration point. With file-based agents, it can be near-empty (discovery handles the rest). With code-based agents, everything is explicitly registered here.

### Full Configuration Shape

```typescript
import { Mastra } from '@mastra/core'
import { Agent } from '@mastra/core/agent'
import { LibSQLStore } from '@mastra/libsql'
import { Memory } from '@mastra/memory'
import { MCPServer } from '@mastra/mcp'
import { Observability, MastraStorageExporter } from '@mastra/observability'
import { PinoLogger } from '@mastra/loggers'

export const mastra = new Mastra({
  // -- Primitives --
  agents: {
    weatherAgent: new Agent({
      id: 'weather-agent',
      name: 'Weather Agent',
      instructions: 'You help with weather.',
      model: 'openai/gpt-5.5',
    }),
  },
  workflows: { /* workflow instances */ },
  tools: { /* shared tool instances */ },
  scorers: { /* scorer instances */ },
  processors: { /* processor instances */ },
  vectors: { /* vector store instances */ },
  tts: { /* voice provider instances */ },
  memory: { /* reusable Memory instances */ },

  // -- Infrastructure --
  storage: new LibSQLStore({
    id: 'mastra-storage',
    url: 'file:./mastra.db',
  }),
  logger: new PinoLogger({ name: 'MyApp', level: 'debug' }),
  observability: new Observability({
    configs: {
      default: {
        serviceName: 'my-app',
        exporters: [new MastraStorageExporter()],
      },
    },
  }),

  // -- MCP Servers --
  mcpServers: {
    myServer: new MCPServer({
      id: 'my-mcp-server',
      name: 'My MCP Server',
      version: '1.0.0',
    }),
  },

  // -- Workspace --
  workspace: new Workspace({
    filesystem: new LocalFilesystem({ basePath: './workspace' }),
  }),

  // -- Background tasks --
  backgroundTasks: {
    enabled: true,
    globalConcurrency: 10,
    defaultTimeoutMs: 300_000,
  },

  // -- Server --
  server: {
    port: 4111,
    host: 'localhost',
    cors: { origin: ['https://example.com'] },
    auth: new MastraJwtAuth({
      secret: process.env.MASTRA_JWT_SECRET,
      mapUserToResourceId: user => user.id,
    }),
    timeout: 180000,
    bodySizeLimit: 4_718_592,
  },

  // -- Bundler --
  bundler: {
    externals: true,
    sourcemap: false,
    transpilePackages: [],
  },
})
```

### Minimal `index.ts` with File-Based Agents Only

When using exclusively file-based agents, the Mastra instance can be minimal (or even empty) because the bundler discovers agents from `src/mastra/agents/*/` directories automatically:

```typescript
import { Mastra } from '@mastra/core'

export const mastra = new Mastra({})
```

## Storage Configuration

Storage is the persistence layer used by memory, workflows, traces, and background tasks. Configured via the `storage` option on `Mastra`.

```typescript
// File-based: src/mastra/storage.ts
import { LibSQLStore } from '@mastra/libsql'

export const storage = new LibSQLStore({
  id: 'mastra-storage',
  url: 'file:./mastra.db',
})
```

Supported backends: LibSQL, PostgreSQL, MongoDB, and more via `@mastra/libsql`, `@mastra/pg`, etc.

## Memory Configuration

Memory gives agents coherence across conversations. Three types:
- **Message history**: Recent messages in a thread
- **Working memory**: Persistent user-specific details (name, preferences)
- **Semantic recall**: Retrieving older messages by relevance

```typescript
// In agent memory.ts (file-based)
import { Memory } from '@mastra/memory'

export const memory = new Memory({
  options: {
    lastMessages: 20,
    workingMemory: { enabled: true },
    semanticRecall: { enabled: true },
  },
})
```

```typescript
// Or code-based, on the Mastra instance
export const mastra = new Mastra({
  storage: new LibSQLStore({ ... }),
  memory: {
    chatMemory: new Memory({
      options: { lastMessages: 20 },
    }),
  },
})
```

## MCP Server Configuration

MCP (Model Context Protocol) servers expose tools, agents, workflows, and resources to MCP-compatible clients.

```typescript
import { Mastra } from '@mastra/core'
import { MCPServer } from '@mastra/mcp'

const mcpServer = new MCPServer({
  id: 'my-mcp-server',
  name: 'My MCP Server',
  version: '1.0.0',
})

export const mastra = new Mastra({
  mcpServers: { myServer: mcpServer },
})
```

For serverless environments, enable stateless MCP:

```typescript
server: {
  mcpOptions: { serverless: true },
}
```

## Discovery Lifecycle

File-based primitives are discovered by the Mastra bundler during `mastra dev` and `mastra build`. Discovery:
- Reads supported files under `src/mastra/`
- Imports TypeScript/JavaScript modules
- Reads markdown instructions and skills
- Copies workspace seed files
- Registers assembled primitives with the Mastra app

**Important**: If you import your `mastra` instance directly (bypassing the CLI), file-based agents are NOT discovered. Use code-based registration in that case.

## Key Findings

1. **Two agent modes coexist**: Code-based (`new Agent(...)`) and file-based (directory convention). File-based is beta but reduces glue code significantly. Mix and match as needed.

2. **File-based minimum**: An agent needs only a `config.ts` and `instructions.md` inside `src/mastra/agents/<name>/`. Start with `mastra dev` and the bundler discovers it.

3. **The `Mastra` class is the central registry**: It holds agents, workflows, tools, memory, storage, MCP servers, observability, and server config. With file-based agents, it can be nearly empty.

4. **Storage is required for memory/background tasks**: Use `LibSQLStore` (file-based SQLite) for development, PostgreSQL/MongoDB for production.

5. **Memory is optional per-agent**: Three types — message history, working memory, semantic recall. Configure via `Memory` from `@mastra/memory`.

6. **MCP servers are independent from agents**: Authored via `MCPServer` from `@mastra/mcp`, registered on the Mastra instance. Expose tools, agents, and workflows to MCP-compatible clients.

7. **Agents vs Workflows**: Agents handle open-ended tasks (LLM decides); workflows handle predetermined multi-step pipelines. Workflows can call agents as steps and vice versa.

8. **CLI is required for file-based discovery**: Running `mastra dev` triggers the bundler's discovery of file-based agents. Direct import of the Mastra instance skips discovery.

9. **Schedules are not yet file-based**: Must be created at runtime via `mastra.schedules.create()`.

10. **Server auto-exposes endpoints**: Agents, workflows, and tools registered on Mastra get REST API endpoints automatically at `http://localhost:4111`. Swagger UI available at `/swagger-ui` when enabled.
